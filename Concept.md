DeepSeek Chat 知识库系统——详细设计文档
版本：1.0
作者：个人知识库项目
日期：2026-06-08
状态：详细设计阶段

## 1. 文档概述
本文档旨在为“外挂大脑”（暂定名）——一个基于 DeepSeek Chat 导出对话的个人知识库系统——提供完整的详细设计。系统定位为本地优先、AI 全自动处理、卡片+图谱双模式的知识管理工具。文档将覆盖需求背景、核心概念、系统架构、模块设计、数据模型、技术选型、部署方案及迭代路线，为后续开发提供清晰蓝图。

## 2. 项目目标与范围
### 2.1 核心目标
- 将 DeepSeek 网页版对话 JSON 自动转化为结构化知识块（卡片）。
- 支持会话-话题双线管理：每个完整对话生成总览卡片，长对话按语义话题切分为独立知识块。
- 通过语义相似度自动关联相似知识点，生成知识图谱，实现浏览和检索。
- 对所有知识块进行版本管理：相同主题的多次讨论形成时间线版本链，保留全部历史。
- 标签采用 LLM 预提取 + 人工定期抽查模式，提供训练反馈闭环。
- 打造卡片式浏览体验，支持全局知识图谱可视化。
- 系统完全由 AI Agent 自动驱动（预处理、分段、总结、标签、关联）。
- 预留扩展接口，未来可导入 Markdown、PDF、网页链接、视频语音等多源内容，并支持安卓端访问 PC 数据库。

### 2.2 范围与边界
- **一期（MVP）**：手动上传 DeepSeek JSON → 自动生成会话摘要+话题卡片 → 存储与语义搜索 → 简版 Streamlit 界面。
- **二期**：完整 Web 卡片界面 + 知识图谱 + 版本管理 + 标签审核训练区 + 外部文档导入（Markdown、PDF、网页）。
- **三期**：安卓客户端（通过 API 访问 PC 服务）、视频/音频转文字、浏览器剪藏插件、间隔复习提醒。

本文档描述整个系统的最终愿景，但会标注各模块的优先级与实施阶段。

## 3. 核心概念模型
| 概念 | 定义 | 示例 |
|------|------|------|
| Session（会话） | 一次完整 DeepSeek 对话导出文件或手动创建的对话记录 | Python多线程与并发-20240601.json |
| Session Card | 会话级摘要卡片，包含对话标题、日期、整体概要、参与角色、知识域分类 | 标题：《Python 多线程最佳实践》；概要：讨论了 GIL、threading 与 asyncio 的适用场景…… |
| Topic Block（话题块） | 从会话中切分出的语义独立片段，是知识的最小原子单位 | 同一对话中切出的“GIL 限制与绕过方法”、“线程池使用案例” |
| Knowledge Card（知识卡片） | 对话题块提炼后的最终知识单元，包含标题、摘要、关键点、代码片段、标签、难度等 | 标题：绕过 GIL 的多线程方案；标签：Python,GIL,多进程；内容：…… |
| Version Chain | 若干语义高度相似的卡片按时间顺序构成的版本链，新版卡指向旧版卡 | 2024-01-15 提问的 pandas groupby → 2024-06-08 同类提问 |
| Tag（标签） | 从知识卡中提取的关键词，经 LLM 建议、用户确认后使用 | pandas, 数据清洗 |
| Category（分类树） | 多层级的领域分类体系，卡片可归属到一个分类节点 | 编程 > Python > 数据科学 |
| Knowledge Graph | 以卡片、会话、标签、分类为节点，关系（包含、版本、关联、标记、属于）为边构成的图 | 局部或全局图谱 |
| External Source（外部源） | 非 DeepSeek 对话的资料：Markdown、PDF、网页、视频等 | 一篇知乎文章、一本电子书笔记 |

## 4. 系统总体架构
系统遵循本地化、离线可用、AI 驱动原则，整体分为四层：

```
┌──────────────────────────────────────────────┐
│                 表示层 (Frontend)             │
│  Web UI (React/Next.js)  /  Streamlit (MVP) │
│    卡片浏览 | 语义搜索 | 图谱可视化 │         │
└──────────────────────────────────────────────┘
                        │ HTTP / WebSocket
┌──────────────────────────────────────────────┐
│              应用服务层 (Backend)             │
│  FastAPI  (REST + GraphQL)                   │
│  - 上传/导入管理                              │
│  - 卡片 CRUD & 版本管理                      │
│  - 图谱查询与渲染数据接口                     │
│  - 人工标注与训练反馈接口                     │
└──────────────────────────────────────────────┘
                        │
┌──────────────────────────────────────────────┐
│                AI 处理管道 (Agent)            │
│  Python 任务引擎 (Prefect / 简单脚本)         │
│  - 预处理 & 话题切分                          │
│  - 卡片生成 (LLM 调用)                        │
│  - 标签提取 & 规范化                          │
│  - 语义向量化 & 相似度去重/关联                │
│  - 图谱关系构建                                │
└──────────────────────────────────────────────┘
                        │
┌──────────────────────────────────────────────┐
│                   数据层                      │
│  SQLite       - 元数据 (会话、卡片、版本、标签)│
│  Chroma/LanceDB - 向量嵌入 (语义搜索)         │
│  KùzuDB       - 知识图谱 (图结构存储与查询)   │
│  文件系统     - 原始 JSON、导入的 PDF/MD 等   │
└──────────────────────────────────────────────┘
```

设计原则：
- 处理管道与 Web 服务解耦，可通过消息队列或简单文件监听触发。
- 所有数据存储于用户指定的本地目录，完全私密。
- 前端采用渐进增强：MVP 用 Streamlit，二期升级为 React 卡片+图谱界面。

## 5. 模块详细设计
### 5.1 数据采集模块
#### 5.1.1 手动上传（一期）
- Web 界面提供拖拽区域，支持单个或多个 .json 文件上传。
- 后端接收后暂存至 uploads/ 目录，记录上传任务到任务队列。
- 校验 JSON 格式，提取基本会话信息（标题、时间、消息数组）。

#### 5.1.2 定时提醒导出
- 系统记录最后导出日期，每日定时（如 20:00）通过系统通知提醒用户“该导出今日的 DeepSeek 对话了”。
- 提醒可配置开关和时间。
- 一期：简单的 Streamlit 页面提示，二期可集成桌面通知（通过 Electron 或浏览器通知）。

#### 5.1.3 外部内容导入（二期）
- Markdown / 纯文本：直接解析为片段，每个二级标题可视为一个话题块。
- PDF：使用 PyMuPDF 提取文本，按页或标题切分。
- 网页链接（知乎等）：通过 Playwright 或 readability 提取正文，转为 Markdown。
- 电子书 (EPUB)：使用 ebooklib 提取章节，按章节切分。
- Bilibili 视频 / 语音（三期）：通过 yt-dlp 下载字幕或音轨，调用 Whisper 语音转文字，再切分处理。
- 所有外部导入内容统一封装为“伪会话”格式进入处理管道。

### 5.2 智能处理管道（AI Agent 工作流）
此管道是系统核心，完全由 LLM 驱动，需设计为可编排的任务流。一期可使用简单 Python 脚本调用 DeepSeek API，二期升级为 Prefect flow。

#### 5.2.1 任务触发
- 上传完成后，后端向任务队列添加一个 `process_session` 任务。
- 工作进程（或同一进程内线程）依次执行以下步骤。

#### 5.2.2 预处理
- 消息清洗：剔除纯表情、无意义单字（“继续”、“哦”）或超短消息（<3 字符）。
- 角色标注：保留 user/assistant 角色，格式化为 `[USER]: ... [ASSISTANT]: ...`。
- 合并连续消息：如果同一角色连续发送多条，合并为一条长消息（便于话题切分）。

#### 5.2.3 会话级总结
调用 LLM，给定整个对话文本，输出 JSON：

```json
{
  "session_title": "优化后的标题",
  "date": "自动解析或使用文件日期",
  "overall_summary": "此对话整体讨论了什么，得到哪些结论",
  "participants": ["user", "DeepSeek"],
  "knowledge_domain": ["编程/Python", "数据科学"]
}
```
生成一条 SessionCard 记录存入数据库。

#### 5.2.4 话题切分
- 方案：使用 LLM 完成切分（而非传统文本分割），因为 LLM 能更好理解语义转折。
- 提示词设计：要求 LLM 在对话流中标记话题变更位置，返回分段索引或直接输出切分后的多段文本。
- 输出格式：

```json
[
  {
    "topic_id": "topic_1",
    "start_msg_index": 0,
    "end_msg_index": 8,
    "topic_title": "GIL限制与解决方案",
    "brief": "讨论了GIL对多线程的影响及绕过方法"
  },
  ...
]
```
每个话题块内可包含多轮 user/assistant 交换。

#### 5.2.5 知识卡片生成
对每个话题块，调用 LLM 生成结构化卡片内容：

```json
{
  "title": "绕过 GIL 的多线程方案",
  "summary": "一句话概括核心",
  "key_points": ["要点1", "要点2", "要点3"],
  "code_snippets": ["代码片段1", "代码片段2"],
  "difficulty": "中级",
  "suggested_tags": ["Python", "GIL", "多线程"],
  "suggested_category": "编程 > Python > 并发",
  "follow_up_questions": ["如果改用协程会怎样？"]
}
```
- 提取 `code_snippets` 时保留原始格式，并标注语言。

#### 5.2.6 标签规范化与去重
- 新卡片 `suggested_tags` 首先与现有标签库进行模糊匹配（如编辑距离、词干匹配），优先复用已有标签。
- 若无可匹配标签，则创建新标签，状态为“待审核”。
- 标签库维护在 SQLite 中，每个标签有名称、使用次数、最后使用日期。

#### 5.2.7 向量化与相似度计算
- 使用本地嵌入模型（如 bge-small-zh 或其他中文模型）将卡片摘要/关键点向量化。
- 计算新卡片向量与库中已有向量的余弦相似度，若高于阈值（如 0.88），则认为属于“相同话题”：
  - 高相似 (≥0.95)：标记为同一话题的新版本，创建 VERSION 关系。
  - 中相似 (0.88~0.95)：标记为 RELATED 关系，等待人工确认（二期可在审核界面确认）。
- 版本链的形成：新卡片的 `parent_version_id` 指向最相似的已有卡片（且相似度符合版本阈值），被指向的卡片若有更早版本，构成链表。
- 所有历史版本保留，卡片详情页展示版本时间线。

#### 5.2.8 图谱关系构建
基于当前任务，在图数据库（Kùzu）中创建节点与边：
- 节点：Session, Card, Tag, Category
- 边：
  - `(Session)-[:CONTAINS]->(Card)`
  - `(Card)-[:TAGGED]->(Tag)`
  - `(Card)-[:BELONGS_TO]->(Category)`
  - `(new_card)-[:VERSION_OF]->(old_card)`
  - `(card1)-[:RELATED]->(card2)` （双向）
- 如果卡片之间有共同的 Tag 超过一定数量，且无直接语义相似，也可加弱关联（可选）。

### 5.3 知识图谱模块
采用嵌入式图数据库 Kùzu（或 Neo4j 社区版）实现。

#### 5.3.1 图模式 (Schema)
```
Node: Session
  - session_id: string
  - title: string
  - created_at: datetime

Node: Card
  - card_id: string
  - title: string
  - summary: string
  - key_points: list[string]
  - code_snippets: list[string]
  - difficulty: string
  - embedding_id: string (关联向量库ID)
  - created_at: datetime

Node: Tag
  - tag_name: string (unique)
  - status: "confirmed" | "suggested"

Node: Category
  - path: string (e.g., "编程/Python/数据科学")
  - level: int

Relations:
  CONTAINS (Session -> Card) : order int
  VERSION_OF (Card -> Card)  : similarity float
  RELATED (Card -> Card)     : similarity float
  TAGGED (Card -> Tag)
  BELONGS_TO (Card -> Category)
```

#### 5.3.2 全局图谱查询
- 默认展示所有节点及其关系，前端采用力导向图布局 (vis-network / D3-force)。
- 性能考虑：
  - 当节点数 > 1000 时，采用“聚焦+展开”策略，初始只显示中心节点（如最新卡片）及其 2-3 层邻居。
  - 提供“全局概览”按钮，采用 WebGL 渲染大规模图（如 three-forcegraph）。
- 节点点击弹出卡片详情面板，支持在图中高亮关联节点。

### 5.4 卡片浏览与交互（前端）
#### 5.4.1 卡片视图（默认）
- 响应式卡片流（Masonry 布局），按时间倒序或相关性排列。
- 每张卡片展示：标题、摘要（前 100 字）、标签条、时间戳、版本指示器（如 “v3” ）。
- 支持过滤：按标签、分类、日期范围、难度。
- 排序选项：最新、最常用、复习频率等。

#### 5.4.2 卡片详情
- 展开完整摘要、关键点列表、代码块（语法高亮，一键复制）。
- 版本时间线：卡片顶部显示横向时间线，列出所有历史版本节点（如 “v1 2024-01-15”, “v2 2024-06-08” ），点击切换显示对应版本的内容。默认显示最新版本。
- 相关卡片推荐列表（基于图谱关系）。
- 操作按钮：编辑标签、修改分类、添加笔记、标记“已复习”。

#### 5.4.3 搜索
- 语义搜索框，输入自然语言，后端通过向量相似度返回最相关卡片。
- 同时支持关键词搜索（全文检索 SQLite 或图谱）。
- 搜索结果显示为卡片列表，可进一步过滤。

#### 5.4.4 图谱视图
- 独立页面，展示知识图谱。
- 可拖拽、缩放，节点可点击。
- 提供“节点类型过滤”（只显示 Card/Tag 等）。
- 点击卡片节点，右侧弹出该卡片的信息摘要。

### 5.5 人工审核与训练反馈
为逐步提高标签和关联质量，设计轻量级人工干预界面。

#### 5.5.1 标签审核区
- 以列表展示所有“待审核”标签及关联的卡片。
- 用户可批量确认、修改、合并或删除标签。
- 修改后的标签作为今后 LLM 提示的示例，不断优化 Prompt（定期重写 system prompt 中的标签列表）。

#### 5.5.2 关联审核区
- 展示 RELATED 关系列表（相似度位于中区间的配对），让用户确认或取消关联。
- 用户的确认/取消可微调后续的相似度阈值，或作为嵌入模型微调的训练样本（未来）。

#### 5.5.3 反馈闭环
- 收集所有人工决策，定期（如每周）自动重新训练一个轻量级分类器（如随机森林），用于辅助预判标签或关联。或者直接将这些决策作为例子加入 LLM prompt 的 few-shot 示例中。

### 5.6 多设备支持方案
- **PC 端（一期/二期）**：Web 应用，本地运行 FastAPI 服务，前端通过浏览器访问 localhost。
- **安卓端（三期）**：开发 Android 应用，通过 REST API 连接 PC 端服务。
  - PC 端需将服务绑定到局域网 IP，并通过防火墙允许手机访问（或通过 Tailscale 组网）。
  - 安卓端功能精简：浏览卡片、搜索、查看图谱简化版。
- 未来可考虑离线同步（客户端本地存储 + 同步协议），但三期先只做“远程访问 PC 数据库”。

## 6. 数据模型详述
### 6.1 关系型数据库 (SQLite)
主要表设计：

```sql
-- 会话表
CREATE TABLE sessions (
    id TEXT PRIMARY KEY,
    title TEXT NOT NULL,
    source_type TEXT DEFAULT 'deepseek',  -- 'deepseek', 'md', 'pdf', etc.
    original_filename TEXT,
    created_date DATE,
    overall_summary TEXT,
    knowledge_domain TEXT,
    metadata_json TEXT,   -- 其他元数据
    uploaded_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 卡片表
CREATE TABLE cards (
    id TEXT PRIMARY KEY,
    session_id TEXT REFERENCES sessions(id),
    title TEXT NOT NULL,
    summary TEXT,
    key_points TEXT,       -- JSON array
    code_snippets TEXT,    -- JSON array
    difficulty TEXT,
    category_path TEXT,
    embedding_id TEXT,     -- 向量库中的ID
    parent_version_id TEXT REFERENCES cards(id) NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP
);

-- 标签表
CREATE TABLE tags (
    name TEXT PRIMARY KEY,
    status TEXT DEFAULT 'suggested',  -- 'suggested', 'confirmed'
    usage_count INTEGER DEFAULT 1
);

-- 卡片-标签关联
CREATE TABLE card_tags (
    card_id TEXT REFERENCES cards(id) ON DELETE CASCADE,
    tag_name TEXT REFERENCES tags(name) ON DELETE CASCADE,
    PRIMARY KEY (card_id, tag_name)
);

-- 外部来源原始数据存储路径（可选）
CREATE TABLE assets (
    id TEXT PRIMARY KEY,
    session_id TEXT,
    file_path TEXT,
    content_hash TEXT
);
```

### 6.2 向量数据库 (Chroma/LanceDB)
存储 `cards.embedding_id` 对应的向量，元数据可存 card_id 便于双向查找。使用相同的嵌入模型保持一致维度。

### 6.3 图数据库 (Kùzu)
如前述图模式，通过 Kùzu 的 Cypher-like 查询语言进行操作。图中节点和关系可与关系型数据一一对应，通过 ID 关联。

## 7. 技术栈选型
| 层次 | 技术 | 理由 |
|------|------|------|
| 后端框架 | FastAPI | 异步支持好，文档自动生成，轻量 |
| AI 管道任务 | Python 脚本 + Prefect (可选) | 灵活编排，易于调试 |
| LLM API | DeepSeek Chat API | 低成本，高质量，与数据源同源 |
| 嵌入模型 | BAAI/bge-small-zh-v1.5 (本地) | 中文语义效果好，本地运行保护隐私 |
| 关系数据库 | SQLite + SQLAlchemy | 零配置，够个人使用，易备份 |
| 向量数据库 | Chroma (嵌入模式) 或 LanceDB | 本地运行，轻量，无需服务 |
| 图数据库 | KùzuDB (嵌入式) | 嵌入式图数据库，无需单独进程，完美适配本地场景 |
| 前端 (MVP) | Streamlit | 极速搭建 UI，适合验证 |
| 前端 (二期) | Next.js + Tailwind CSS + vis-network/D3 | 卡片、图谱高度定制 |
| 桌面集成 | 二期考虑用 Electron 打包，或保留 Web | 便于后台提醒、文件拖拽 |
| 安卓端 | Kotlin + Retrofit + 简单的 Jetpack Compose UI | 原生体验，仅调用 REST API |
| 文件解析 | PyMuPDF, ebooklib, readability-lxml, yt-dlp, whisper | 成熟库，覆盖各种格式 |
| 部署 | 本地 localhost，或树莓派 + Tailscale | 个人可控，随时访问 |

## 8. 部署方案
### 8.1 开发环境
- 所有服务运行在同一台 PC。
- Python 环境管理使用 Conda 或 venv。
- 前端开发时 `npm run dev` 即可。

### 8.2 生产部署（个人使用）
- 后端 + AI 管道：使用 systemd (Linux) 或 launchd (macOS) 注册为服务，开机自启。
- 前端 Next.js 可 `next start` 长期运行，或直接用 Electron 打包成桌面应用，内嵌 WebView，减少端口占用。
- 数据目录可自定义，默认 `~/ai_knowledge_base/`。
- 备份策略：SQLite 和向量库文件简单压缩定时备份。

### 8.3 安卓访问
- PC 端服务监听 0.0.0.0:8000，通过路由器固定 IP 或 Tailscale 分配固定虚拟 IP。
- 安卓 App 配置服务器地址，使用 Token 简单鉴权（本地局域网可忽略安全性，或通过 HTTP Basic Auth）。

## 9. 迭代路线图
### Phase 1: MVP (0-4 周)
- 上传 DeepSeek JSON → 预处理 → 会话总结 → 话题切分 → 知识卡片生成（Streamlit）
- 存储到 SQLite + Chroma
- 语义搜索命令行/Streamlit 页面
- 不包含：图谱、版本链、标签审核、外部导入

### Phase 2: 完整个人知识库 (4-10 周)
- 迁移前端至 Next.js，实现卡片流、详情、版本时间线
- 接入 KùzuDB，构建知识图谱并可视化
- 版本管理（相似度自动关联，时间线切换）
- 标签审核与训练反馈界面
- 外部导入支持：Markdown, PDF, 网页
- 定时导出提醒（桌面通知）
- 全局图谱（初始展开，可缩略图）

### Phase 3: 扩展与多端 (10-16 周)
- 安卓客户端开发（浏览、搜索）
- 视频/音频导入与转文字
- 浏览器剪藏插件
- 间隔重复复习提醒
- 用户行为分析，持续优化 Agent 提示

## 10. 关键挑战与应对
| 挑战 | 应对策略 |
|------|----------|
| LLM 分段不准确 | 允许人工调整分段，提供修正界面（二期）；不断优化提示词，加入少量示例 |
| 标签爆炸与不一致 | 严格的标签归一化（相似度合并），定期人工审核；提供标签库管理工具 |
| 向量相似度阈值设定 | 初期采用经验值，人工审核区收集正负样本，调整阈值或训练分类器 |
| 全局图谱性能 | 使用 Kùzu 的投影查询，前端限定邻居深度；大规模时采用 WebGL 渲染 |
| 外部内容质量参差不齐 | 导入时允许用户预览和筛选，AI 处理前提供片段确认（可选） |
| DeepSeek 导出 JSON 格式变更 | 设计健壮的解析器，支持多版本 schema，用户可反馈异常 |
| 隐私与合规 | 所有数据存储于本地，不上传至任何云端；LLM 调用仅传输处理所需的文本片段 |

## 11. 附录：LLM 提示词设计原则
- **结构化输出**：始终要求返回 JSON，使用 `response_format` 控制。
- **角色设定**：“你是一个专业的个人知识管理助手，擅长总结、分类和发现知识关联。”
- **上下文长度控制**：对于长对话，先让 LLM 分段，再对每个分段单独总结，避免超长上下文失真。
- **温度设置**：总结和标签提取使用较低温度 (0.2~0.3)，确保一致性。
- **反馈学习**：将用户修正后的标签作为 future few-shot 示例注入提示词。

## 二、扩展端集成改造

只需在原扩展基础上增加 3 个轻量模块，不破坏现有功能。

### 2.1 新增消息类型与通信层

在 `background/message-router.ts` 中新增：

| 消息类型                | 方向                      | 功能                                                 |
| ----------------------- | ------------------------- | ---------------------------------------------------- |
| `PUSH_TO_SERVICE`       | Content / Manager → BG    | 将抓取的 session 数据发送到本地服务                  |
| `QUERY_SERVICE`         | Manager → BG              | 代理请求本地服务 API（卡片列表、搜索等）             |
| `CHECK_SERVICE_STATUS`  | Manager → BG              | 检查本地服务是否在线                                 |

Service Worker 中使用 `fetch` 与 `http://localhost:8000` 通信，设置 3 秒超时。

### 2.2 抓取流程增强

在内容脚本的“Topic Selector”弹窗中增加一个选项：

> “同时推送到本地知识库” （默认勾选）

用户确认后，扩展先按原逻辑存入 IndexedDB，再调用 `PUSH_TO_SERVICE` 将原始消息数组 + 元数据发送至本地服务。

若本地服务不可达，显示一条 Toast 提示，并将推送任务记录到本地队列（可重试）。

### 2.3 Manager UI 轻量改造

- 侧边栏增加入口：“本地知识库” → 若服务在线，展示最近 10 张卡片（列表 + 摘要），点击跳转独立 Web 管理界面。
- 设置页面：配置本地服务地址（默认 `http://localhost:8000`），可开启/关闭自动推送。
- 保留原有主题管理功能，让用户可仅使用扩展进行轻量整理（适合不想启动服务的场景）。

## 三、本地服务端集成适配

在知识库系统设计中增加一个 **扩展专用 API 模块**，与原有上传接口兼容。

### 3.1 新增 API 端点

```python
# POST /api/v1/extension/sessions
{
  "session_id": "uuid (扩展生成的)",
  "title": "来自抓取",
  "source_url": "https://chat.deepseek.com/...",
  "messages": [
    {"role": "user", "content": "...", "timestamp": "..."},
    {"role": "assistant", "content": "...", "timestamp": "..."}
  ],
  "metadata": {...}
}
```

处理逻辑：

1. 将接收的数据转换为内部 `Session` 对象（与上传 JSON 文件格式统一）。
2. 加入任务队列，触发完整的 AI 处理管道（总结 → 切分 → 卡片 → 标签 → 向量化 → 图谱更新）。
3. 返回处理任务 ID，前端可轮询进度。

### 3.2 兼容原有设计

- 知识库系统的所有核心模块（数据模型、AI 管道、向量库、图数据库）无需修改。
- 仅需在数据采集模块增加一个“扩展推送”的适配层，将扩展推送的 `messages[]` 转换为内部统一格式即可。
- 定时提醒导出功能（知识库设计中的 5.1.2）可改为通过扩展的 `chrome.alarms` 触发，避免依赖桌面通知。

### 3.3 服务发现与状态

- 服务启动时在本地临时文件写入端口号，扩展通过读取该文件或轮询 `http://localhost:8000/health` 判断状态。
- 服务提供简易 WebSocket 接口（如 `ws://localhost:8000/ws/task/{task_id}`），让扩展实时获取卡片生成进度。

## 四、协同工作流举例

**场景：抓取并自动生成知识卡片**

1. 用户浏览 DeepSeek 对话 → 点击扩展按钮“抓取 + 推送”。
2. 扩展发送完整对话到本地服务，收到 `task_id: abc`。
3. 扩展 Manager 侧边栏的“本地知识库”小面板显示 “处理中… 预计 30 秒”。
4. 本地服务后台：
   - 调用 LLM 生成会话摘要
   - 切分出 3 个话题块
   - 为每个块生成知识卡片，提取标签
   - 向量化后与现有卡片做相似度计算，发现有一张与历史卡片相似度 0.92，自动建立版本链
   - 更新图谱
5. 扩展轮询到任务完成，弹出通知：“已生成 3 张新卡片，点击查看”。
6. 点击跳转到独立 Web 管理界面，直接定位到新卡片。

## 五、方案优势

- **保留知识库全部设计**：本地服务完全按照你的设计文档实现，无阉割。
- **扩展体验零破坏**：现有抓取、话题管理功能可继续使用；推送是可选增强。
- **渐进式开发**：一期仍可先做 Streamlit MVP + 手动上传 JSON；扩展集成作为二期功能加入，不影响整体进度。
- **离线与隐私**：所有数据与处理均在本地，扩展仅作为本地通信前端，无外部依赖。
- **未来可扩展**：独立 Web 前端可独立升级，安卓端直接连接本地服务，扩展无需大改。

## 六、实施建议

1. 先启动本地知识库服务开发（Phase 1 MVP），实现上传 JSON 和核心 AI 管道。
2. 在服务稳定后，扩展端增加“推送到本地服务”功能（大约增加 200 行 TS 代码）。
3. 逐步将扩展的 Manager 部分高级功能（如语义搜索、图谱）委托给本地服务 API，扩展仅做轻量展示，保持其轻快特性。
4. 最终形成“扩展采集 + 本地服务智能处理 + Web 管理界面全功能浏览”的铁三角。
